# Palate WhatsApp Phase 1 Backend

Lean FastAPI backend for Palate WhatsApp verification, verified phone capture, session/order mapping, WhatsApp message sending, payment webhook handling, and basic operational tracking.

This project is intentionally a backend/integration layer. Palate continues to own the web app screens, menu, cart, payment page, review page, and production hosting.

Final handover, go-live support boundary, and Phase 2 separation are documented in [`PHASE1_FINAL_HANDOVER_AND_SUPPORT.md`](./PHASE1_FINAL_HANDOVER_AND_SUPPORT.md).

## Current Scope

Included:

- Health/readiness endpoints
- WhatsApp session-link creation
- Secure `PALATE_XXXXXXXX` token generation
- Token hashing with server-side pepper
- WhatsApp webhook verification for Meta Cloud API
- Twilio WhatsApp adapter for testing or fallback provider use
- Verified WhatsApp sender phone capture
- Customer/session/order/restaurant mapping
- First-session welcome/menu WhatsApp message after verification
- One short separate first-session follow-up message after the menu link
- Order summary, bill/payment, payment success/failure, and feedback/review message helpers
- Razorpay webhook signature verification and event dedupe
- Internal API key protection for private endpoints
- Basic signed-in operational dashboard
- Docker/Alembic/PostgreSQL deployment package

Not included:

- Admin dashboard
- Vendor dashboard
- Full frontend screens
- Loyalty/offers/campaign systems
- AI chatbot or AI agent
- Native WhatsApp Payments
- Complex analytics
- Marketing automation

## What The WhatsApp Flow Does

1. Palate screen calls `POST /api/v1/whatsapp/session-link`.
2. Backend creates a secure token and stores only its hash.
3. Backend returns `session_id`, `wa_url`, and expiry.
4. User opens WhatsApp and sends the prefilled message.
5. WhatsApp webhook receives the real sender phone and message body.
6. Backend extracts the `PALATE_XXXXXXXX` token and matches the stored hash.
7. Backend links phone, customer, session, restaurant, and order context.
8. Backend marks the session verified.
9. Backend sends the welcome/menu or order-context message.
10. Palate polls `GET /api/v1/whatsapp/sessions/{session_id}` and saves `verified_phone`.

## Exact First-Session Messages

Incoming user message generated by the session link:

```text
Start my Palate session and show me <Restaurant>'s menu
Token: PALATE_XXXXXXXX
```

First backend reply after successful verification on menu/landing flow:

```text
Welcome to Palate.
Here is <Restaurant>'s menu. Tap any dish to leave a review:
<menu_url>
```

Separate short follow-up message:

```text
One thing that makes us different: you rate the dish, not the restaurant.
Because one place can have a brilliant dal and a forgettable paneer, and you deserve to know which is which.
```

No long timed onboarding automation sequence is part of this Phase 1 package.

## Create WhatsApp Session

```http
POST /api/v1/whatsapp/session-link
```

```json
{
  "restaurant_id": "rest_123",
  "restaurant_name": "Burger House",
  "browser_session_id": "browser_abc_123",
  "entry_point": "menu",
  "intent": "verify_phone",
  "resume_url": "https://development-vendors.palatepower.com/menu/dalchini"
}
```

Response:

```json
{
  "session_id": "uuid",
  "wa_url": "https://wa.me/...",
  "expires_at": "2026-05-12T10:00:00Z"
}
```

For best UX, Palate should create this session in the background when the screen context is ready. The button should only open the returned `wa_url`.

  "can_proceed": true,

  "next_action": "resume_flow",

## Dynamic Order, Payment, And Review Links

Palate remains the source of truth for menu, cart, bill, payment, and review screens. This backend does not create those pages. It accepts the final dynamic URLs from Palate, stores them against the order context, wraps them in tracked links, and sends them on WhatsApp.

Supported dynamic fields on `POST /api/v1/captain/orders`:

```text
menu_url
order_url
bill_url
payment_url
feedback_url
dish_reviews[].review_url
```

The same endpoint is an upsert by `external_order_id`, so Palate can first create the order with partial context and later call it again with payment/review URLs when those become available.

Message endpoints use the latest stored order context:

```text
POST /api/v1/orders/{order_id}/send-whatsapp-summary
POST /api/v1/orders/{order_id}/send-bill
POST /api/v1/orders/{order_id}/send-feedback
```

Razorpay webhooks map back to orders through:

```json
{
  "notes": {
    "order_id": "<palate_whatsapp_order_uuid>",
    "external_order_id": "<palate_external_order_id>"
  }
}
```

If a matching order has a verified WhatsApp phone, captured/paid Razorpay events send a payment-success WhatsApp message as a best-effort utility update. Failed payment events send a payment-pending message when a retry link exists.

## Provider Switch

Use one provider at runtime:

```text
WHATSAPP_PROVIDER=meta
WHATSAPP_PROVIDER=twilio
WHATSAPP_PROVIDER=mock
```

Production default should be:

```text
WHATSAPP_PROVIDER=meta
DEMO_MODE=false
META_MOCK_MODE=false
META_SEND_ENABLED=true
```

Twilio can be used for testing or as a fallback provider:

```text
WHATSAPP_PROVIDER=twilio
TWILIO_WEBHOOK_AUTH_ENABLED=true
```

## Security Notes

- Session tokens are never stored raw.
- Webhooks are verified where provider signatures are available.
- Webhook events are deduped.
- Private APIs require `X-API-Key`.
- Dashboard requires password login.
- Secrets must come from environment variables only.
- Razorpay phone/contact data is treated as payment metadata, not verified Palate identity.

## Current Palate-Side Pending Items

Palate needs to wire its current screens to these endpoints and provide final dynamic URLs from its own app:

- Menu/QR screen creates `session-link`
- Cart/payment screen creates `session-link` if user is not verified
- Invoice/bill creation calls `captain/orders` or bill send endpoint when invoice URL exists
- Review flow sends per-dish review URLs when available
- Payment/Razorpay mapping can be completed once Palate payment route is finalized

The backend is ready to receive these contexts. Palate owns the screen placement and final URLs.
