# Lead-OS AI Sales Ecosystem

This document serves as the permanent architectural reference for the Lead-OS project. It defines the three core "Machines" (Agents), how they interact, their deployment status, and the standard configuration approach.

## The Three Machines

### 🤖 Machine 1: The Intelligence Engine (`lead_os_v2.py`)
- **Role:** Autonomous Lead Generation & Deep Research.
- **Function:** Scrapes Google Maps (via Apify) to find clinics, then deep-crawls their websites (via Firecrawl/Steel.dev) to analyze their text, finding owner names, services offered, and determining if they are a good fit. It generates highly personalized icebreakers and scores the leads.
- **Output:** Saves highly enriched lead data to JSON files in the `output/` directory and writes them to Google Sheets/Databases.
- **Status:** Completed and functional locally. Can be run via scheduled batch scripts.

### 🤖 Machine 2: The Acquisition Engine (`sales_engine.py`)
- **Role:** Outbound Campaign Manager (Cold Outreach).
- **Function:** Reads the verified, enriched leads generated by Machine 1. It orchestrates outbound campaigns by sending the initial Cold Emails (via Resend) and Initial WhatsApp messages (via Twilio Meta Cloud API). It tracks status in a local SQLite database (`lead_queue.db`).
- **Status:** Completed and functional locally. Can be run on a schedule to process a set number of leads per day.

### 🤖 Machine 3: The Omnichannel Sales Closer (`leados_sales/` & `api.py`)
- **Role:** Inbound Conversation Handler (The Closer).
- **Function:** When a lead replies to Machine 2's outreach (via Email or WhatsApp), Machine 3 takes over. It is a live, always-on server (`api.py`) that receives Twilio Webhooks. It uses advanced LangGraph routing to hold natural conversations, handle objections, and ultimately push the lead to book a demo via Cal.com/Calendly.
- **Status:** Deployed to Railway. The webhook is actively listening at `https://<YOUR_RAILWAY_DOMAIN>/webhook/whatsapp`.

## Deployment & Configuration

### One Environment File to Rule Them All (`.env`)
All three machines share exactly **one** `.env` file located at the root of the project. This ensures that no matter which agent is running, they share the exact same credentials.

**Core Required Environment Variables:**
- `OPENAI_API_KEY`: Powers the core reasoning for all 3 machines.
- `TWILIO_ACCOUNT_SID`, `TWILIO_AUTH_TOKEN`, `TWILIO_PHONE_NUMBER`: Used by Machine 2 for outbound, and Machine 3 for inbound WhatsApp.
- `RESEND_API_KEY`: Used by Machine 2 for cold emails.
- `APIFY_API_TOKEN`: Used by Machine 1 for Map scraping.
- `FIRECRAWL_API_KEY` / `STEEL_API_KEY`: Used by Machine 1 for deep website text analysis.
- `BOOKING_LINK`: Used by Machine 3 to close the sale (e.g., `https://cal.com/yourname/demo`).

### Where Do They Live?
- **Machine 1 & 2** are designed to run as **cron jobs or scheduled tasks**. Because they process large batches of data, running them locally or on a dedicated background worker is best. They do not need to be exposed to the internet.
- **Machine 3** is a **Live Server** deployed on **Railway**. It must be available 24/7 to receive instant Webhook HTTP POST requests from Twilio the second a lead replies.

### Current Database Architecture
Currently, state is managed via two main methods:
1. `output/` directory JSON files (for Machine 1 -> Machine 2 handoff).
2. `lead_queue.db` (SQLite) to track who has been messaged, who has replied, and who booked a demo. 

*Note: As the project scales to 10,000+ clinics, migrating the SQLite database and JSON files into a unified PostgreSQL database (like Supabase) is the recommended Phase 2 upgrade.*

## Taking Over (For Future AI Agents)

If you are an AI agent reading this, you are working on the **Lead-OS** project. 
- Ignore the `AI-AGENTS-NEW` and `zrai-lead-os-full` directories—they are deprecated legacy systems.
- Focus exclusively on:
  - `lead_os_v2.py`
  - `sales_engine.py`
  - `api.py` (Webhook root)
  - `leados_sales/` (LangGraph memory/closer logic)

If Twilio WhatsApp replies are failing, always check the `api.py` FastAPI server deployed on Railway and ensure the Twilio Console Webhook URL is pointing to `https://<RAILWAY_DOMAIN>/webhook/whatsapp` for incoming messages.
